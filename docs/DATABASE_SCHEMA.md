# Re-MIND-eЯ — Database Schema Reference

This document serves as the schema reference for all active PostgreSQL tables, views, custom functions, and triggers in the Re-MIND-eЯ application.

---

## 1. Tables

### profiles
Stores core user details linked to Supabase Auth.
- `id` (UUID, PK): References `auth.users(id)`.
- `full_name` (Text): Display name.
- `telegram_chat_id` (Text, Unique): Linked Telegram ID.
- `role` (Text): Role flag (`PATIENT` or `CAREGIVER`).
- `phone_number` (Text): Contact number.
- `created_at` (Timestamp).

### medications
Defines active medications and schedules for patient Telegram accounts.
- `id` (BigInt, PK).
- `telegram_id` (Text): References patient's `telegram_chat_id`.
- `drug_name` (Text).
- `dosage` (Text).
- `frequency` (Text): e.g., `once_daily`, `twice_daily`, `thrice_daily`.
- `tablet_count` (Integer): Stock tracking count.
- `low_stock_alert_enabled` (Boolean).
- `reminder_times` (Text[]): Time strings (e.g., `['08:00', '20:00']`).
- `priority_level` (Text): `normal` or `critical`.
- `active` (Boolean).
- `timezone` (Text): Medication's scheduled timezone (e.g., `Asia/Kolkata`).

### reminder_events
Dose reminders generated by the scheduler or dynamic client rules.
- `id` (BigInt, PK).
- `medication_id` (BigInt): References `medications(id)`.
- `telegram_id` (Text): References patient's `telegram_chat_id`.
- `scheduled_for` (Timestamp).
- `reminder_status` (Text): `PENDING_PATIENT`, `TAKEN`, `SKIPPED`, `ESCALATED_TO_CG`, `UNCONFIRMED`.
- `snooze_count` (Integer).

### reminder_logs
Historical adherence log entries for compliance charting.
- `id` (BigInt, PK).
- `telegram_id` (Text).
- `medication_id` (BigInt): References `medications(id)`.
- `scheduled_time` (Timestamp).
- `response` (Text): `TAKEN`, `SKIP`, `MISSED`.
- `created_at` (Timestamp).

### notifications
System alerts displayed in the UI notification bell.
- `id` (BigInt, PK).
- `user_id` (UUID): Recipient's profile ID.
- `title` (Text).
- `message` (Text).
- `type` (Text): `CARE_CIRCLE_ACCESS_REQUEST`, `CARE_CIRCLE_ACCESS_GRANTED`, etc.
- `connection_id` (UUID): Optional connection link context.
- `created_at` (Timestamp).

### caregiver_connections
Many-to-many relationship tracking.
- `id` (UUID, PK).
- `patient_profile_id` (UUID): References `profiles(id)`.
- `caregiver_profile_id` (UUID): References `profiles(id)`.
- `connection_status` (Text): `PENDING`, `ACCEPTED`, `REJECTED`.
- `is_active` (Boolean).
- `relationship_type` (Text).
- `is_primary` (Boolean).
- `can_view_medications` (Boolean).
- `can_view_vault` (Boolean).
- `can_view_reports` (Boolean).
- `can_edit_medications` (Boolean).
- `can_receive_escalations` (Boolean).

---

## 2. Views

### active_caregiver_links
Unifies `caregiver_connections` (modern many-to-many table) and `caregiver_info` (legacy registry table) into a single, compatibility read-only view. Used by bot and background worker to resolve relationship permissions.

---

## 3. Custom Functions & Triggers

- **`handle_caregiver_connection_trust_events()`**: Fires on INSERT/UPDATE in `caregiver_connections` to automatically audit changes and insert access request alerts into `notifications` (executed as `SECURITY DEFINER`).
- **`handle_reminder_event_state_change()`**: Fires on INSERT/UPDATE of `reminder_events`. Generates notification alerts and loops through all active caregivers to fan out alerts on escalations or confirmations.
